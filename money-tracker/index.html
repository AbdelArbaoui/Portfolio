Pourquoi le choix du Local-First ?

Les applications financières traditionnelles imposent de stocker vos données bancaires sur des serveurs tiers cloud. Cela engendre des risques de sécurité majeurs (fuites de données, piratage), des abonnements récurrents onéreux et rend l'application dépendante d'une connexion internet.

Money Tracker résout ces problèmes en adoptant un modèle local-first strict.

cancel Modèle standard Cloud

  • • Serveur cloud stockant vos comptes
  • • Identifiants de banques partagés en ligne
  • • Modèle SaaS payant obligatoire
  • • Hors-ligne inutilisable

check_circle Modèle Local-First

  • • SQLite local chiffré AES-128
  • • Clés stockées dans le Keychain OS
  • • 100% gratuit & à vie
  • • Mode offline natif ultra-rapide

Inspection de la Base SQLite

Mode chiffré

Toutes les colonnes critiques (IBAN, soldes bruts, clés d'accès API) sont chiffrées en base via **Fernet (AES-128)**. Le code Python utilise des accesseurs transparents qui ne déchiffrent qu'à la volée en mémoire vive.

Compte Courant Boursorama gAAAAABmX3K... gAAAAABmY8L...
Livret A Caisse d'Epargne gAAAAABmP2D... gAAAAABmD9V...
lock

État de la table : Chiffré de bout en bout

Les valeurs réelles ne sont visibles par aucun processus système autre que Money Tracker. En cas d'extraction brute du fichier `db.sqlite3`, le contenu reste totalement indéchiffrable.

code

Code Backend (SQLAlchemy + keyring)

class BankAccount(Base):
    __tablename__ = "bank_accounts"

    # Stockage de la colonne chiffrée
    balance_enc = Column(String, nullable=True)

    @property
    def balance(self) -> float:
        if not self.balance_enc:
            return 0.0
        return float(decrypt_data(self.balance_enc))

    @balance.setter
    def balance(self, value: float):
        self.balance_enc = encrypt_data(str(value))

Zéro clé API en clair : Le secret maître Fernet est lui-même extrait au démarrage du Keychain OS natif via la bibliothèque `keyring`. Le mot de passe n'apparaît ni dans les commits, ni dans les fichiers `.env`.

lock_reset

Simulateur de Verrou Mutex

SQLite n'autorise qu'un seul écrivain à la fois. Pour éliminer les erreurs `database is locked` lors des synchronisations automatiques multiples, un **Mutex unique asynchrone (`asyncio.Lock`)** orchestre l'accès au moteur de sync.

État du Mutex _SYNC_LOCK
lock_open Libre (Unlocked)
Tâche A : Sync Banque DSP2 Idle
Tâche B : Sync Trade Republic Idle
verified_user

Pourquoi cette mécanique est robuste ?

  • timer
    Évite les conflits d'écritures : La seconde tâche attend sagement que la première tâche ait fini de valider ses transactions SQLite avant d'acquérir le verrou.
  • crisis_alert
    Timeout et résilience de 60s : Si un connecteur externe plante (DSP2 en panne, OTP expiré), la tâche lève une exception sécurisée et libère le verrou (`finally: lock.release()`) pour ne pas bloquer les synchronisations suivantes.
  • shield
    Cooldown restrictif de 60s : Le backend rejette les requêtes de sync trop rapprochées dans le temps pour économiser l'usage réseau et éviter les bannissements d'IP (rate limit).

Behavior-Driven Model Agent Development (bMAD)

La méthodologie **bMAD** est un cadre d'ingénierie rigoureux qui structure le développement collaboratif entre des agents d'IA autonomes et l'humain. Elle repose sur des spécifications formalisées permettant de supprimer les régressions.

Phase 1

PRD & Specs

Spécification stricte de 64 exigences fonctionnelles dans `prd.md` comme unique source de vérité.

Phase 2

ADR Design

Consensus sur 10 Architectural Decision Records (ADR) archivés dans `architecture.md`.

Phase 3

Sprints Roadmaps

Backlog de sprint découpé en Epics et suivi de tâches structuré (`sprint-status.yaml`).

Phase 4

Agent Autonome

Exécution par un agent autonome avec validation systématique des tests de non-régression.

Phase 1 : PRD & Spécifications fonctionnelles strictes

Avant d'écrire la première ligne de code, l'ensemble des règles de gestion financière, des calculs d'abonnements, de résilience locale et de synchronisation d'investissements ont été fixés dans un document de PRD. Cette préparation limite les hallucinations et garantit la cohérence du développement d'IA.
